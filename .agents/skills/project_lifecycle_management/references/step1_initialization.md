# Step 1 详细执行标准：项目工程物理初始化与多 Agent 协作规范

> [!NOTE]
> 本规范为项目生命周期 Step 1 的具体执行细则，旨在确立标准的工程物理结构与 Agent 间的协作边界。

---

## 一、 标准目录结构初始化 (SOP)

### 1. 物理结构划分
在仓库根目录下，必须明确划分以下三个核心模块：
*   `frontend/`：前端模块目录。
*   `backend/`：后端模块目录。
*   `docs/`：项目规范文档目录，作为整个项目的**“真理之源（Source of Truth）”**。

### 2. `docs/` 的 11 个分层子目录定义
在 `docs/` 目录下，必须按顺序创建以下 11 个分层子目录。此排序代表了标准的软件工程设计与文档流转顺序：

| 序号 | 目录名称 | 归属领域 | 职责说明 |
| :--- | :--- | :--- | :--- |
| 01 | `docs/01_business_research/` | 业务问题域 | 业务背景与方案调研 |
| 02 | `docs/02_competitor_analysis/` | 业务问题域 | 竞品与对标方案分析 |
| 03 | `docs/03_business_modeling/` | 业务问题域 | 业务问题建模 |
| 04 | `docs/04_interaction_design/` | 核心交互与体验 | 交互行为链路设计 |
| 05 | `docs/05_ux_specification/` | 交互与协同实现 | 界面与表现标准规范 |
| 06 | `docs/06_system_architecture/` | 系统工程域 | 系统整体架构设计 |
| 07 | `docs/07_data_model/` | 系统工程域 | 数据结构与数据模型设计 |
| 08 | `docs/08_api_specification/` | 系统工程域 | 通信协议与接口契约规范 |

| 11 | `docs/11_integration_and_deployment/` | 工程实现域 | 系统集成联调与发布部署方案 |

### 3. 占位与追踪约定

> [!IMPORTANT]
> **空目录约束**：
> * **绝对禁止预置大纲模板**。各目录在初始化阶段必须保持为空。
> * 为了防止版本控制系统忽略空目录，必须在各核心层级目录及规范层子目录中分别创建一个空的 `.gitkeep` 占位文件。

## 二、 多 Agent 职责与协作边界

为了确保在多智能体自主开发模式下，开发流的强内聚与松耦合，必须在项目根目录下生成协作边界契约 [AGENTS.md](../../../AGENTS.md)，并硬性约束以下协作机制：

### 1. 真理之源（Source of Truth）原则
*   **先规范，后代码**：凡是涉及业务实体、通信协议、数据结构或系统架构的设计变更，**严禁直接修改代码**。
*   **流转机制**：必须首先在规范层目录（如 `docs/`）的相应设计规范中提出修改并通过审查后，负责代码实现的智能体才能进行工程开发。

### 2. 前后端强解耦自治

> [!WARNING]
> **目录读写限制**：
> * **前端限制**：前端 Agent 仅允许读写 `frontend/` 目录下的文件。严禁读写 `backend/` 下的任何代码。
> * **后端限制**：后端 Agent 仅允许读写 `backend/` 目录下的文件。严禁读写 `frontend/` 下的任何代码。
> * **唯一契约**：前端与后端的唯一联调和协议桥梁是 `docs/08_api_specification/` 中定义的 API 规范。
