# Step 2 详细执行标准：引导用户添加项目级别 Rule 过程

> [!NOTE]
> 本规范旨在指导 Agent 如何在项目生命周期中引导人类用户确立并维护项目专有的协作与技术规则 (Project-Scoped Rules)，以确保多 Agent 协作和技术方案的一致性。

---

## 一、 引导机制与触发时机

*   **项目物理初始化完成后**：在完成 Step 1 项目工程初始化与协作边界对接后，应当主动询问人类用户是否存在项目特有的技术架构选型、模块与环境管理方式、配置读写规范或研发流程管理标准。
*   **日常迭代发现**：在日常开发或方案设计中，若发现多次重复的技术设计与决策（如外部模块引入规则、系统参数读取机制、前置流程审查契约等），应当引导人类用户将其沉淀为项目级别的 Rule。

---

## 二、 规则持久化要求

*   **写入位置**：项目级别的规则必须统一持久化到项目根目录下的 [AGENTS.md](../../../AGENTS.md) 文件中。
*   **合并原则**：若 [AGENTS.md](../../../AGENTS.md) 已存在，应将新规则追加至合适的内容区块，避免直接覆盖已有规则。若文件不存在，应新建并初始化该文件。
*   **配套文件变更**：若规则确立或变更涉及到通用项目配置文件（如版本控制忽略规则文件、编辑器格式配置文件等）的修改，应将其作为配套变更一并列出并提交。

---

## 三、 引导交互步骤与规范

> [!IMPORTANT]
> **绝对禁令**：
> 严禁在未与用户交互确认的情况下，擅自将推测的或默认的工程规范直接写入项目配置文件中。

AI 必须严格执行以下四步交互流程：

### 1. 主动对话询问
在项目初始化完成后，AI 必须主动发起对话，以列表形式列出常见的规范维度（如外部依赖管理机制、配置加载机制、工作流契约规范等），向用户征求意见或推荐最佳实践，供用户进行选择与补充。

### 2. 制定与修改草案
根据用户在对话中确认的具体诉求，将规则整理为条理清晰的 Markdown 条目草案。若该规则涉及工程配置文件的配套变更（例如在引入特定的包管理器或构建工具时，需要修改版本控制忽略规则以过滤相关缓存或生成物），应同步生成对应文件的修改草案。

### 3. 提请审核与配套变更确认
向人类用户展示完整的修改方案，包含 [AGENTS.md](../../../AGENTS.md) 规则草案以及所有涉及的通用配置文件（如版本控制忽略规则文件）的配套变更对比。

### 4. 写入与变更
**在获取人类用户明确的审核通过后**，方可执行写入与变更操作。
