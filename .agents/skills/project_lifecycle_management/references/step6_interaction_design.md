# Step 6 详细执行标准：核心交互链路设计与状态规范

> [!NOTE]
> 本规范为项目生命周期 Step 6 的通用执行细则，旨在定义系统交互链路、多角色交互时序以及异常与边界处理的标准化设计方法论，指导交互链路设计的输出并确保其与业务模型精确对齐。

> 2. **物理产出路径**：交互设计的收口规范文档必须归档至项目根目录下的 `docs/04_interaction_design/` 目录中，推荐命名为 `flow_state_spec-v1.0.md`。

## 二、 交互设计核心要素与方法论

设计人员（智能体或人类）在执行核心交互链路设计时，必须在收口规范文档中详细推导并输出以下核心要素：

### 1. 核心实体状态机规范 (State Machine Specification)
必须定义系统主要实体的生命周期状态机，以明确前后端数据流转与页面交互响应契约：
*   **状态节点划分**：梳理实体生命周期的全部状态，涵盖初始化、活跃、挂起/休眠、结束/归档等核心阶段。
*   **转移触发矩阵**：以明晰的表格形式列明状态转移的源状态、目标状态、触发事件（如用户操作、定时任务、系统通知等）及对应的系统行为。
*   **状态图可视化**：必须为核心实体绘制 Mermaid 格式的状态图，清晰展示其生命周期闭环与转移路径。

### 2. 多角色交互时序规范 (Sequence Diagrams)
为界定用户、前端界面、后端服务及外部协作模块或受限沙箱之间的交互边界，**必须绘制 Mermaid 格式的多角色交互时序图**。时序设计需重点覆盖：
*   **数据采集与编译器触发流程**：梳理多通道数据抓取、请求发送、底层编译、校验存储的完整处理时序。
*   **逻辑拓扑与校验阻断流程**：梳理关系建立、本地/服务端即时拓扑校验、异常阻断状态触发以及手动干预解环的处理时序。

### 3. 核心交互链路详细设计 (Interactive Loops)
交互设计文档应深入探讨系统关键业务闭环的页面流转与异常处理细节，分为以下四类通用场景：

#### 3.1 实体创建与初始化交互
*   **输入表单与字段约束**：详述新建实体时的页面流转与表单项配置。定义字段的校验约束与强限制（如边界值校验）。
*   **依赖与绑定机制**：定义实体创建时的前置要求（如强制关联外部媒介）及相关子系统或受限任务模块的静默绑定机制。

#### 3.2 信息解析、渲染与展现交互
*   **多维度渲染与分栏布局**：设计满足沉浸式体验的界面分栏布局。针对长文本或多级结构化数据，必须设计渐进式加载（如骨架屏）以降低感知延迟。
*   **多通道协作交互**：详细定义不同视图/分栏之间的联动机制，如触发机制的弱提示气泡、悬浮菜单以及跨区域的数据流注入。
*   **数据源缓存与闲时增量同步**：针对高计算开销或高成本的同步操作，必须设计手动触发/闲时异步同步的交互，并辅以图谱等物理可视化组件，实现概念节点的关联定位与高亮脉冲定位。

#### 3.3 可视化编辑与逻辑关系变更交互
*   **可视化卡片画布编辑器**：定义节点卡片的拖拽排序、连接线绘制以及前置/后继依赖关系的交互逻辑。支持无依赖节点的并行设计。
*   **冲突与依赖环路阻断交互**：定义逻辑关系产生冲突（如环路死锁）时的系统级阻断行为。要求通过界面高亮、红色发光与抖动等视觉动效进行强警示，提供清晰的成环路径警报，并强制禁用提交流程（如置灰批准按钮），直至用户手动解除冲突。

#### 3.4 计划执行与异常调度交互
*   **逾期冲突与挂起重调度**：定义长周期执行过程中出现节点逾期或调度冲突时的异常状态表现（如高亮变红、闪烁警示）。设计重调度交互面板，提供“一键依赖链联动顺延”与“甘特图/日历手动拖拽微调”双轨方案，在解决冲突的同时保障非依赖流的独立操作性。
*   **超时会话休眠与状态重载**：详述系统长周期会话的优雅断开与休眠机制，状态持久化至持久层（如缓存/数据库）。在用户重登时提供非阻塞的视觉弱提示（如毛玻璃遮罩），并通过一键重载操作，配合重载动效，快速恢复上下文。

## 三、 角色职责与协作机制

在核心交互链路设计生命周期中，参与者应遵循以下协作边界进行推进：

### 1. 交互设计智能体/设计人员 职责
* **状态与链路定义**：基于业务模型，拆解交互链路状态机，设计多角色协作时序图，编写详细设计规范。
* **边界与体验打磨**：对各交互阶段的视觉提示、阻断动效与异常流进行细致设计，确保与前置契约一致。

### 2. 审查智能体/技术评审人员 职责
* **物理边界与安全契约审计**：严格校验交互规范是否对核心技术红线与安全隔离契约在 UI/交互层面进行了完整的闭环闭锁。
* **异常交互覆盖率审计**：审查所有极端和边缘状态（如网络丢包、逻辑死锁、操作超时）的页面容错设计，防范交互死锁。

### 3. 人类用户 职责
* **体验确认**：审查智能体提交的交互链路与状态规范。确认通过后，解封后续系统架构设计、数据模型与 API 规范的细化设计工作。

## 四、 成果产出标准规范

核心交互链路设计最终必须在 `docs/04_interaction_design/` 目录下产出唯一的收口规范：
* **《[项目名称]交互链路与状态规范》** (`flow_state_spec-v1.0.md`)

报告必须严格包含以下结构，严禁留空或使用占位符：
*   **一、 核心实体状态机规范**：包含主要实体的 Mermaid 状态图与状态流转行为契约表。
*   **二、 角色互动时序图**：包含至少两个核心交互闭环的 Mermaid 多角色互动时序图。
*   **三、 核心交互链路详细设计**：按实体创建、信息解析渲染、可视化编辑及计划执行四大部分详细说明页面流转、交互动效、校验规则及异常容错设计。
*   **四、 关键契约与边界设计汇总表**：归纳系统架构与技术选型在交互层面的限制契约（如闲时同步、环路阻断、休眠重载、安全隔离），列出具体交互响应要求与物理限制。

## 五、 输出与排版标准

* **相对路径引用**：文档内所有涉及前置文档或同级文件的超链接，必须使用合规的**相对路径**，严禁使用绝对路径。
* **无 Emoji 限制**：所有文档及报告正文与标题中严禁包含任何 Emoji 图标，确保文档的专业度与系统兼容性。
* **排版美观度**：深度应用 Markdown 语法，多用表格、Mermaid 状态与时序图表进行描述，并利用 GitHub 警示框（如 `> [!IMPORTANT]`、`> [!CAUTION]`、`> [!WARNING]`）来突出设计契约红线与异常阻断交互。
