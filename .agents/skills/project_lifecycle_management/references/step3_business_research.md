# Step 3 详细执行标准：业务调研与反向审查规范

> [!NOTE]
> 本规范为项目生命周期 Step 3 的具体执行细则，旨在指导 Agent 在项目开发前如何系统地开展正向业务调研与反向审查，确立健壮的项目业务底座与机制设计。

---

## 一、 执行顺序约束铁律

> [!IMPORTANT]
> **串行执行约束与自启动要求**：
> 在开展业务调研时，必须严格遵循“先正向成文，后反向审查”的串行顺序与自启动规则：
> 1.  **正向调研成文**：首先必须完整执行 **正向业务调研**，将所有分析结果（场景定位、受众画像、选型对比、全链路图）写入项目根目录下的 `docs/01_business_research/business_research.md` 文件中，并向人类用户提请审核。
> 2.  **自动启动反向审查**：一旦人类用户对正向调研报告给出审核通过的明确反馈（如“通过”、“同意”或“OK”等），智能体必须**立即自动、无缝地启动反向审查与防御设计**，严禁停滞等待人类用户二次指派任务或发布催促指令。
> 3.  **基于成文文档启动反向审查**：Agent 必须**且仅能基于已成文的正向报告**，启动 **反向审查与防御设计**。严禁在正向报告尚未定稿前，一边设计一边挑刺或并行混淆这两个阶段。

---

## 二、 正向业务调研规范

在进入代码开发前，Agent 必须以“真理之源（Source of Truth）”为导向，与人类用户进行充分沟通，明确业务场景的核心定位与技术方案路线：

### 1. 引导机制与交互 SOP
*   **推荐 /grill-me 访谈命令**：对于新项目的业务背景对齐，应当引导用户运行 `/grill-me` 交互式命令，通过多轮问答厘清设计边界。
*   **单步提问原则**：在交互式采访中，Agent 必须使用 `ask_question` 工具向用户提问。**每次回复仅允许提出一个核心决策问题**，并提供清晰的推荐选项和依据，禁止一次性抛出多个问题。
*   **禁止单方面臆断**：严禁在未与用户交互确认的情况下，擅自推测或硬编码业务场景的核心定位和技术选型偏好。

### 2. 正向调研四大核心要素
调研报告必须以清晰的结构覆盖以下四个核心板块：
*   **场景目标定义**：明确系统的核心业务定位，诊断传统模式存在的核心痛点，并推导其产生的深层原因。
*   **目标人群画像**：定义系统的首要目标受众和次要受众，总结其习惯、技术背景和关键诉求。
*   **技术选型对比与决策**：列出至少 2-3 种备选架构或技术框架方案，从开发成本、状态控制机制、人机交互复杂度等维度进行客观对比，并基于业务流程特征给出最终的选型依据。
*   **全链路业务流程设计**：绘制核心业务流程图（可采用规范的图示语言），必须清晰展示数据输入流向、核心处理节点、分支流转条件以及人机协同（如挂起点）的挂起位置。对流程图中的每一个核心业务阶段进行详细的功能和处理逻辑描述。

## 三、 反向审查与防御规范

在完成正向调研后，Agent 必须扮演“魔鬼代言人（Devil's Advocate）”角色，对正向调研结论进行批判性审视，识别潜在漏洞并补充防御性对策。

### 1. 多角度反向审查核心维度

Agent 进行审查时，必须分别从**业务与体验**、**技术与架构**、**安全与合规**三个角度进行深度审视，并挑战以下高频漏洞：

*   **业务与体验侧反向审查（Business & Experience Review）**：
    *   **过度工程化（Over-engineering）**：正向设计中是否有非必要的复杂业务逻辑？是否可以用更简单的逻辑、静态规则或轻量级流程代替？是否增加了用户无端的交互与心智负担？
    *   **运行成本与可行性（Cost & Feasibility）**：系统运行的资源与计算成本（如 API 调用、Token 消耗、算力等）是否合理可控？在商业/运行成本上是否具备长期可持续性？
    *   **体验延迟与流控（UX Latency & Flow Control）**：长链路或外部大模型服务调用的响应延迟是否会打断用户的操作流畅度，造成不可接受的体验？是否有合理的挂起、异步或提示机制？
*   **技术与架构侧反向审查（Technology & Architecture Review）**：
    *   **极端边界与异常恢复（Edge Cases & Recovery）**：若系统遇到空值、非预期格式、越界输入，或发生外部依赖调用中断、并发冲突，状态机如何优雅重置、回滚或报错？如何保证数据一致性？
    *   **技术债与扩展瓶颈（Scalability & Tech Debt）**：选定的架构与技术方案是否存在已知缺陷、理论性能上限、扩展瓶颈或数据失真风险？
    *   **人机协同与挂起设计（Human-in-the-loop Constraints）**：人机交互中的挂起与恢复设计是否稳健？当人类长时间未响应时，系统状态是否会陷入死锁或无限挂起？
*   **安全与合规侧反向审查（Security & Compliance Review）**：
    *   **系统安全与特权限制（System Security）**：系统及其调用的外部服务是否被赋予了过高的执行特权？如何防范非法的恶意注入（如 Prompt 注入）与指令越权执行风险？
    *   **隐私与合规红线（Privacy & Compliance）**：系统处理的敏感数据、知识资产或组织机密信息是否存在泄露风险？系统整体是否具备合规或政策性隐患？

### 2. 业务与技术难点评估矩阵（Difficulty Matrix）

为了量化评估识别出的风险与难点，Agent 必须构建一个二维的**难点评估矩阵**。评估矩阵包含以下两个维度：

*   **业务影响度（Business Impact）**：
    *   **高**：直接影响核心体验、系统安全、合规性，或存在不可接受的延迟，导致业务流程无法闭环。
    *   **中**：影响非核心功能或次要用户体验，有可接受的临时绕过方案。
    *   **低**：对核心业务闭环无实质影响，属于锦上添花或边缘体验优化。
*   **实现复杂度（Implementation Complexity）**：
    *   **高**：涉及复杂的算法研发、架构重构、深度的第三方系统集成或高风险的并发/状态一致性设计。
    *   **中**：涉及常规的业务模块开发、标准接口集成或具有一定复杂度的交互逻辑。
    *   **低**：常规增删改查、简单的静态规则配置或样式调整。

评估矩阵的评估结果须以表格形式展现，并划分为以下四个象限，采取不同的应对策略：

| 业务影响度 \ 实现复杂度 | 低复杂度 | 高复杂度 |
| :--- | :--- | :--- |
| **高影响度** | **【防守要塞】**：属于核心防御重点。应通过明确的规范与常规技术手段（如校验、重试、流控）直接落定。 | **【关键决战】**：核心技术难点与主要风险源。**必须提供详细的多方案“决策建议”**，交由 Lead 评审裁决。 |
| **低影响度** | **【日常迭代】**：纳入正常开发排期，不作为特殊待审批项。 | **【过度工程】**：属于范围蔓延。建议果断精简、砍掉该设计或移出首期 MVP 范围。 |

### 3. 多方案决策建议与折中机制（Decision Recommendations）

针对评估矩阵中落入**【关键决战】**（高影响-高复杂度）象限的核心难点，Agent 必须提供结构化的**决策建议**，通过对比折中方案（Trade-offs）辅助人类进行技术/业务决策。每个决策建议须包含：

*   **难点/风险项编号与描述**：标明难点并指明其在矩阵中的位置。
*   **方案A（推荐方案）**：
    *   **实现逻辑**：概述如何技术落地。
    *   **优势（Pros）**：对业务、性能或安全的长期收益。
    *   **缺点与代价（Cons）**：开发成本、资源消耗或引入的次要限制。
*   **方案B（替代方案）**：
    *   **实现逻辑**：概述替代路径（通常是更轻量或规避难点的方式）。
    *   **优势（Pros）**：如低开发成本、高响应速度等。
    *   **缺点与代价（Cons）**：如业务覆盖度不足、可能存在体验损失等。
*   **决策建议与折中分析（Trade-off Analysis）**：详细分析为什么推荐方案A，解释团队必须接受何种折中代价。

### 4. 防御性对策闭环机制与待审批项输出

*   **输出为待审批项（Pending Approvals）**：反向审查的直接产出必须为结构化的**待审批项清单**，包含**难点评估矩阵**与针对高危难点的**决策建议明细**。
*   **自动追加与提交闭环**：一旦触发自启动审查，智能体必须在当轮交互中将这些待审批项以 `## 六、 反向审查与防御设计` 为标题追加写入到项目根目录下的 `docs/01_business_research/business_research_adversarial.md` 文件中。
*   **交互确认与落定**：智能体应引导人类用户对这些待审批项进行逐一审批或交互式对齐（如通过 `/grill-me`）。在用户确认通过后，智能体必须将确认落定的“防御性对策”和“具体落地技术手段”更新写入到报告及 Walkthrough 中，形成单次交互内的完整自执行闭环。
*   **对策可落地性**：落定的防御性对策必须有清晰的软件工程对应手段（如输入输出校验机制、并发控制、异常处理逻辑等），禁止使用无法量化的软性约束。

## 四、 专业 Lead 评审与业务总结指引

在正向业务调研与反向审查（包括待审批项）均成文后，必须由 Agent 扮演“专业 Lead”角色对调研成果与风险防御进行评审和最终技术裁决。

本阶段的 Lead 评审执行标准遵循通用的 **[Lead 评审参考标准](./lead_review.md)**。

### 1. 评审前置依赖与双文档输入

> [!IMPORTANT]
> **多源信息输入与深度研判约束**：
> 在执行评审与输出总结报告前，扮演“专业 Lead”的 Agent **必须同时完整阅读并综合研判** 以下两份独立成文的原始文档。严禁仅凭单方报告、自行臆断或在文档不全的情况下擅自开展评审：
> 1.  **正向业务调研报告**：[business_research.md](../../../../docs/01_business_research/business_research.md)（承载业务场景定义、目标人群画像、技术选型对比及全链路业务流程图等正向设计事实）。
> 2.  **反向审查与防御报告**：[business_research_adversarial.md](../../../../docs/01_business_research/business_research_adversarial.md)（承载多角度反向审查出的风险漏洞、难点评估矩阵、多方案决策建议及待审批项等反向博弈证据）。

### 2. 本阶段特定输出定位与汇总逻辑

*   **汇总决策文档物理文件名**：必须在综合研判上述两份输入文档并完成评审后，生成唯一的阶段汇总总结报告：[business_summary.md](../../../../docs/01_business_research/business_summary.md)。
*   **逻辑定位说明**：[business_summary.md](../../../../docs/01_business_research/business_summary.md) **并非单纯的总结性/汇报性报告**。它是对正向调研报告的研发边界做出的最终裁决，同时也是对反向审查报告中的待审批项（PA）与多方案折中做出的最终决策。它作为**后续业务问题域各阶段（如 02.竞品分析、03.业务问题建模）的前置输入与设计底座**。后续步骤均须在本文档所限定的核心业务领域范围和技术防线约束下开展开发与设计。

### 3. 业务调研阶段特化前置总结文档模板

[business_summary.md](../../../../docs/01_business_research/business_summary.md) 必须按照通用规范进行业务调研层面的特异化，其内容应覆盖：
*   **一、 Lead 整体评审结论**：对调研范围给出明确的宏观定调（例如：“批准通过，立项进入设计阶段”）。
*   **二、 核心业务领域与业务边界裁决**：明晰界定哪些能力与场景纳入本系统的核心业务领域范围（如：解析带读、KI 提炼和异步安全技能生成），哪些特性不属于核心领域或排除在业务边界之外（如：多人协作、多模态 OCR 等），用于规范后续设计与实施范围。
*   **三、 待审批项（PA）裁决明细**：针对反向审查提出的 **PA-01**（结合难点矩阵与决策建议中的方案 A/B 折中选择），进行最终的技术路线裁决，详述落地的技术手段与架构理由，确立后续技术实现的底座契约。
*   **四、 核心业务闭环与技术防线**：总结系统构筑的安全、输入、运行三道物理技术防线。
*   **五、 后续任务与工程指引**：明确说明向后续业务问题域及技术架构域（如启动数据模型设计与 API 规范制定）推进的指引。

## 五、 输出与排版标准

*   **正向业务调研报告写入位置**：写入项目根目录下的 [business_research.md](../../../../docs/01_business_research/business_research.md) 文件中。
*   **反向审查与防御报告写入位置**：写入项目根目录下的 [business_research_adversarial.md](../../../../docs/01_business_research/business_research_adversarial.md) 文件中。
*   **业务前置总结文档写入位置**：Lead 审批与技术裁决的总结文档统一写入项目根目录下的 [business_summary.md](../../../../docs/01_business_research/business_summary.md) 文件中。
*   **排版规范**：所有文档必须使用标准的 Markdown 语法，严禁在文中包含任何 Emoji 图标。多使用列表、表格与 GitHub 警示框（如 `> [!IMPORTANT]`、`> [!NOTE]`）来突出关键的决策边界和技术红线。所有文件路径引用必须使用相对路径。
