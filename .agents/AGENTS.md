# 多 Agent 职责与协作边界

> [!NOTE]
> 本文档定义了在多 Agent 开发模式下，前端与后端智能体的核心职责边界与协作机制，以确保开发流的强内聚与松耦合。

---

## 一、 协作机制与铁律

### 1. 真理之源（Source of Truth）原则

> [!IMPORTANT]
> **先规范，后代码**：
> * 凡是涉及业务实体、API 协议、数据库表结构或系统架构的设计变更，**严禁直接修改代码**。
> * **流转机制**：必须首先在 [docs/](../docs/) 目录的相应规范中提出修改并通过审查后，代码级 Agent 才能进行开发实现。

### 2. 前后端强解耦自治

> [!WARNING]
> **目录读写限制**：
> * **前端限制**：前端 Agent 仅允许读写 [frontend/](../frontend/) 目录下的文件。严禁读写 [backend/](../backend/) 下的任何代码。
> * **后端限制**：后端 Agent 仅允许读写 [backend/](../backend/) 目录下的文件。严禁读写 [frontend/](../frontend/) 下的任何代码。
> * **唯一契约**：前端与后端的唯一联调和协议桥梁是 [docs/08_api_specification/](../docs/08_api_specification/) 中定义的 API 规范。

---

## 二、 项目级别规则 (Project-Scoped Rules)

### 1. 项目管理与环境配置
* **项目管理工具**：使用 `uc` 进行虚拟项目管理。
* **环境配置**：所有本地开发配置与敏感变量必须存放在项目根目录下的 `.env` 文件中。

### 2. 研发工作流与规划
* **复杂任务优先 `/grill-me`**：对于复杂的架构设计、核心逻辑重构或不确定性较高的任务，智能体应当优先推荐用户运行 `/grill-me` 交互式命令，通过访谈对齐设计方案。
* **先规划后执行**：所有非微调性质的开发任务，必须严格执行“构思方案 → 提请审核 → 制定实施计划与任务列表 → 用户确认通过”的步骤，规划确认后方可进入代码实现阶段。
