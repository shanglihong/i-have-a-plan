# 决策总结与技术裁决前置文档

> [!IMPORTANT]
> 本文档是由专业 Lead 对前期的设计/调研主文档进行评审与裁决后生成的最终总结，确立了当前阶段研发的业务边界与核心技术路线。本文档为后续详细设计、竞品分析、业务建模与工程实现的唯一前置输入底座与真理之源（Source of Truth）。

---

## 一、 概览

本阶段对辅助阅读与知识技能沉淀系统（基于“项目”底座的计划辅助系统）进行了系统的业务调研与安全防御审视，评估的原始输入文档如下：

1. **正向业务调研报告**：[business_research.md](./business_research.md)
   * **内容简述**：系统化定义了以“项目（Project）”为底座的统一业务领域模型，特化出“计划项目”与“阅读项目”；提出了“AI辅导章节任务链”与双向驱动的伴读交互模式；确立了基于向量/图谱混合知识库和 Trace-to-Skill 的技术框架大图。
2. **反向审查与防御报告**：[business_research_adversarial.md](./business_research_adversarial.md)
   * **内容简述**：扮演“魔鬼代言人”对上述方案进行安全和体验审查，诊断出 Graph RAG 运行成本高昂、Trace-to-Skill 编译幻觉、外部文本注入提权越权、以及硬性流程打扰用户沉浸感等高危风险，并输出待审批决策矩阵。

---

## 二、 核心关键点共识

开发团队与 Lead 就系统的高阶模型与交互设计达成以下核心共识，作为后续系统架构与研发设计的底层契约：

1. **项目统一化底座 (Project Foundation)**：一切用户任务均以“项目”实体承载，特色的“阅读项目”是项目的一种特化实现，在底层复用项目生命周期管理。
2. **AI 辅导与章节任务链 (AI Mentoring)**：阅读项目不是无约束伴读，而是结合书籍/论文实际章节和方法论 Markdown 文件，自动生成结构化的“章节阅读任务链”，由阅读辅助 Agent 扮演导师，以“双向驱动”交互进行启发式辅导。
3. **Skill 人机友好编译规范 (Skill Representation)**：Trace-to-Skill 编译出的方法论技能采用类似于 `SKILL.md` 的结构化 Markdown 文件存储（含 YAML 元数据与 Markdown 步骤指引），便于人类阅读查阅和 Agent 解析执行。
4. **技能反哺匹配 (Semantic Skill Injection)**：在普通计划项目创建时，计划辅导 Agent 通过对项目目标进行语义匹配，主动向用户推荐技能库中沉淀的 Skill，一键载入用于指导和生成专业的任务列表。

---

## 三、 核心分歧与风险

经正反双向文档研判，系统在架构设计与商业落地中存在以下三大核心风险：

1. **运行成本风险 (PA-02)**：Graph RAG 的图谱构建与检索涉及超高频的大模型调用，在长文本电子书与密集文献场景下，会产生巨大的 Token 运营成本。
2. **执行死锁风险 (PA-03)**：Trace-to-Skill 编译非结构化方法论时易产生大模型幻觉，若生成的 Skill 步骤存在逻辑死锁（循环依赖）或缺失关键条件，会导致后续计划生成器崩溃。
3. **特权注入风险 (PA-05)**：外部上传的书籍、论文属于非安全数据源，文档中可能暗含恶意的 Prompt 注入指令，若伴读 Agent 拥有敏感系统调用特权，可能被诱导执行非授权指令导致数据泄露。

## 四、 审查对象

明确本阶段审查和落定的具体实体与业务边界，用于规范开发与设计的首期范围：

* **纳入开发与设计范围的审查对象**：
  * 项目实体（Project）分类与状态机模型。
  * 融合方法论的章节阅读任务链生成机制。
  * 伴读 Session 轨迹提取器与 Trace-to-Skill 编译器沙箱。
  * 轻量级增量知识图谱（Graph RAG）与向量库（Dense RAG）双轨索引。
  * 计划辅导 Agent 的语义 Skill 检索与“引导挂起-校验”推进器。
* **排除出本周期设计边界的对象**：
  * 第三方笔记/云盘数据源自动同步（微信读书、Obsidian 笔记库自动导入等），MVP 首期仅支持本地上传及 Zotero 导入。
  * 无人类干预的自动 Skill 上线与后台静默执行，必须保证人类控制权。

## 五、 决策建议

针对第三章的核心风险与分歧，由 Lead 给出最终的技术裁决与防御建议：

1. **Graph RAG 运行成本裁决 (PA-02)**：
   * **决策建议**：采纳 **方案 A (异步增量与按需提取图谱)**。
   * **技术理由与防线**：日常伴读仅做轻量向量缓存（Dense RAG）。只有在用户手动点选“生成图网”或阅读项目被标记为“读完/阶段归档”时，才在后台启动异步批处理提取合并图谱关系。以此节约 80% Token 消耗，保障低延迟。
2. **Trace-to-Skill 编译可靠性裁决 (PA-03)**：
   * **决策建议**：采纳 **方案 A (技能沙箱与人类审批门控 - Human-in-the-loop)**。
   * **技术理由与防线**：编译出的 Skill 必须先隔离存储在沙箱，生成直观任务链草稿。由用户进行步骤和依赖的微调，手动点击“批准入库”方可归档进正式 Skill Store。这既筑牢了逻辑防线，又是用户巩固所学知识的优良交互体验。
3. **Prompt 注入安全防御裁决 (PA-05)**：
   * **决策建议**：采纳 **方案 A (物理特权隔离与极小化 API 权限)**。
   * **技术理由与防线**：从物理运行时沙箱进行权限收紧。伴读 Agent 沙箱不具备任何文件写入、系统 Shell 执行和外部网络外连的 API 特权，仅有只读文本和对话框输出权限，彻底阻断注入指令的危害路径。
