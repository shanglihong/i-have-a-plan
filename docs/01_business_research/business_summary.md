## 二、 核心业务领域与业务边界裁决

为避免项目范围蔓延，保障 MVP 首期研发的聚焦与稳定，对业务边界做出如下硬性界定：

### 1. 纳入核心业务领域（In-Scope）
* **双特化项目管理**：支持通用的“计划项目”与特定特化的“阅读项目”的创建、状态跟踪与生命周期流转。
* **双轨知识检索与构建**：实现 Dense RAG（细节局部检索）与轻量级 Graph RAG（跨书籍网状概念关系联想）协同，支持个人第二大脑可视化图网构建。
* **双向驱动 AI 辅导**：章节阅读任务链的智能拆解，支持用户主动提问，并在核心方法论章节由阅读辅助 Agent 主动发起启发式辅导问答。
* **Trace-to-Skill 沙箱编译**：在阅读项目伴读中自动捕获方法论，在技能沙箱中编译为结构化 `SKILL.md`，由人类进行微调确认并入库。
* **计划推荐与渐进执行**：计划辅助 Agent 基于项目描述进行语义匹配，主动向用户推荐沉淀的 Skill，以“挂起-校验”渐进推进计划。

### 2. 排除出业务边界（Out-of-Scope）
* **多渠道自动同步接口**：首期不提供与 Notion、微信读书、Obsidian 等第三方云端笔记库的双向同步，仅支持本地文件（PDF、EPUB、Markdown）上传和 Zotero 文献导入。
* **全自动无监管 Skill 执行**：系统不允许 Agent 在无人类确认（Human-in-the-loop）的情况下直接自动注入并执行技能，确保执行链条的安全受控。

## 三、 待审批项 (PA) 裁决明细

针对反向审查报告中提出的三个【关键决战】难点，经 Lead 评审，做出最终的技术与架构裁决：

| 决策项编号 | 决策描述 | 最终锁定方案 | 技术裁决依据与落地手段 |
| :--- | :--- | :--- | :--- |
| **PA-02** | Graph RAG 构建引发的高额 Token 消耗成本 | **方案 A**：异步增量与按需提取图谱。 | **依据**：个人用户对图谱关联无秒级实时性要求。<br>**落地手段**：日常阅读仅将笔记、问答存入向量库（Dense RAG）。只有在用户手动触发“合并图谱”或该阅读项目“归档/阶段性完成”时，才在后台启动异步批处理（Batch）提取并合并实体关系，节约 80% Token 成本。 |
| **PA-03** | Trace-to-Skill 提炼方法论产生的幻觉与逻辑失效 | **方案 A**：技能沙箱与人类审批门控（Human-in-the-loop）。 | **依据**：大模型逻辑幻觉是目前技术难以完全解决的，不能容忍损坏的技能直接注入生产计划。<br>**落地手段**：编译出的 Skill 进入“待审核沙箱”，前端生成直观任务链，用户（人类）进行步骤与依赖的微调，手动点选“批准入库”方可生效。这也是知识内化的好时机。 |
| **PA-05** | 不可信上传文档中的恶意 Prompt 注入与越权指令 | **方案 A**：物理特权隔离与极小化 API 权限沙箱。 | **依据**：系统与用户数据的安全性是第一红线。<br>**落地手段**：运行物理隔离。伴读 Agent 的运行上下文不被赋予任何文件修改、网络外连或敏感指令执行特权，仅有当前文本读取权及对话输出权，使 Prompt 注入行为在物理上失去利用空间。 |

## 四、 核心业务闭环与技术防线

为了保障系统运行的健壮性，技术架构中必须构筑以下三道防线：

* **安全防线 (物理特权隔离)**：伴读/辅导 Agent 处于完全无特权的运行时沙箱中。任何从文档解析出的恶意注入指令，由于沙箱无 API 权限支持，均无法提权对系统造成危害。
* **输入/技能防线 (单元校验与人类审查)**：Trace-to-Skill 产生的 `SKILL.md` 必须通过内部 JSON Schema 语法检测以防损坏，并最终经由人类在沙箱中进行内容和逻辑审批，杜绝死锁 Skill 流入生产库。
* **运行防线 (异步流控与延迟批处理)**：图谱构建由独立批处理队列管理，限制单次和单位时间内的 API 调用频率，防止突发流量引发外部 LLM 提供商限流（Rate Limit）导致服务崩溃。
