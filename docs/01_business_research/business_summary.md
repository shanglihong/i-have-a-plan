## 二、 核心关键点共识与边界界定

开发团队与 Lead 就系统的高阶模型与交互设计达成以下核心共识，作为后续系统架构与研发设计的底层契约：

- **1. 项目统一化底座 (Project Foundation)**
  > 确立一切用户任务以“项目”实体承载的底座逻辑。特色的“阅读项目”是项目的一种特化类型，在底层完全复用项目生命周期管理。
- **2. AI 辅导与章节任务链 (AI Mentoring)**
  > 拒绝无约束随性伴读。系统融合书籍/论文目录与阅读方法论自动生成结构化的“AI辅导章节任务链”，由阅读辅助 Agent 扮演导师，以双向驱动方式开展启发式辅导。
- **3. Skill 人机友好编译规范 (Skill Representation)**
  > Trace-to-Skill 编译出的方法论技能采用类似于 `SKILL.md` 的结构化 Markdown 文件存储（含 YAML 元数据与 Markdown 步骤指引），既方便人类直接查阅修改，又便于 Agent 解析执行。
- **4. 技能反哺匹配 (Semantic Skill Injection)**
  > 当用户创建普通计划项目时，计划辅导 Agent 通过对项目目标和描述进行语义检索匹配，主动向用户推荐技能库中沉淀 of Skill，一键载入用于指导和生成专业的任务列表。

## 四、 待审批项 (PA) 技术裁决明细 (决策分级)

针对反向审查与防御性分析中暴露出的核心难点，根据风险和复杂度进行分级裁决，最终状态清零：

### 1. P1 级裁决 (Lead 自决项)

#### 待审批项 PA-02：混合知识库 (Graph RAG) 构建的巨额运行与 Token 成本
- **背景与难点**：每次划线或伴读时实时调用 LLM 进行知识实体与关系提取，会导致极高的运行延迟与高昂的 API 调用成本。
- **技术裁决方案**：建立“向量缓存 + 后台异步批处理建图”机制。日常伴读只进行密集向量检索（Dense RAG）并写入临时缓存，用于基本的章节上下文问答。
- **折中与依据**：**平时伴读阅读仅做向量缓存（Dense RAG）**。仅在项目归档或用户主动触发时，由后台启动异步批处理生成合并实体图谱。图谱更新存在非实时延迟，用户在刚读完新章节时无法在思维图网上面即时查阅到联想。
- **当前状态**：**已裁决并锁定**。

#### 待审批项 PA-05：伴读 Agent 的特权注入与命令越权执行风险
- **背景与难点**：外部导入的电子书/文献中可能携带恶意 Prompt 注入指令。若伴读 Agent 拥有敏感系统调用特权，易被注入指令越权。
- **技术裁决方案**：建立“物理特权隔离极小化 API 权限沙箱”机制。
- **折中与依据**：从物理运行时沙箱进行权限收紧。伴读 Agent 沙箱不具备任何文件写入、系统 Shell 执行和外部网络外连的 API 特权，仅保留只读文本和对话框输出权限。伴读 Agent 丧失了调用外连工具查阅的特权能力，以安全性换取隔离度。
- **当前状态**：**已裁决并锁定**。

### 2. P2 级裁决 (人机共决项)

#### 待审批项 PA-03：大模型提炼方法论技能 (Trace-to-Skill) 时的幻觉与依赖死锁
- **背景与难点**：Trace-to-Skill 提炼非结构化方法论时模型易产生幻觉，生成的 Skill 步骤包含循环依赖或空参数，会导致后续任务拆解崩溃。
- **技术裁决方案**：引入“技能沙箱审批与人类审批门控（Human-in-the-loop）”机制。
- **折中与依据**：提炼出的 Skill 临时隔离于沙箱中，系统生成可视化任务链。增加了用户（人类）对步骤、参数和依赖进行微调和确认勾选批准的操作，确保在进入全局技能库前幻觉已被纠正。此决策已升级，最终同意以这一交互步骤作为“人机协同知识内化”的关键契约。
- **当前状态**：**已通过升级机制由人类用户在交互中授权落定**。
