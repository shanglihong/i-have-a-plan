# 前后端功能边界与通信协议规范 v1.0

> [!IMPORTANT]
> 本文档基于 [《业务模型规范》](../03_business_modeling/business_model.md) 与 [《交互链路与状态规范》](../04_interaction_design/flow_state_spec-v1.0.md) 梳理，定义了前端系统与后端混合服务之间的职责边界、数据供给关系及通信协议契约。所有 API 开发与状态机流转必须以此文档为真理之源（Source of Truth）。

## 一、 系统职责与边界划分

本系统采用“厚前端、强中台”的架构思想。为了保障交互的极低延迟和沉浸体验，部分传统的校验职责下放至前端，而后端专注于重型运算、数据持久化与大模型编排。

### 1. 前端专属职责 (Frontend Responsibilities)
- **拓扑排序与环路阻断**：沙箱编辑器内的卡片连线合法性校验（是否形成环路），**必须**由前端在本地内存中运行拓扑排序算法实现，以提供毫秒级的红色抖动阻断动效。仅在校验通过后才向后端发起保存/入库请求。
- ** Trace-to-Source 模糊匹配容错**：根据笔记绑定的首尾特征字符，在页面 DOM 中寻找最相似的物理段落。此逻辑在前端利用 JS 执行相似度计算并定位，不请求后端。
- **UI 状态机维护与防抖机制**：
  - 富文本笔记输入的 500ms 自动防抖保存。
  - 阅读进度条的双维度加权计算（结合组件内部状态与后端返回的切片进度）。
  - 水波纹唤醒动画、平滑滚动、脉冲闪烁的高帧率渲染控制。
- **本地缓存脱敏 (PA-06)**：个人思考卡片在内存中的临时加密存储管理。

### 2. 后端专属职责 (Backend Responsibilities)
- **业务领域模型持久化**：操作数据库（PostgreSQL/MySQL 等）存储项目、任务、笔记和技能等核心实体的最终状态。
- ** Agent 生命周期的物理隔离管控 (PA-05)**：为每个伴读/监督 Agent 开启独立受限沙箱进程。执行严格的权限控制，剥夺其系统网络与文件写入权限。
- **会话超时与休眠控制 (PA-04)**：后端守护进程（Daemon）严格统计会话活跃度，24小时无交互自动将大模型会话 Context 序列化至 Redis，释放物理连接，并向前端推送休眠信号。
- **后台闲时图谱构建 (PA-02)**：
  - 核心计算任务。通过定时任务或闲时队列，异步调用大模型进行实体和关系（Graph RAG）的抽取，操作图数据库合并节点。

## 二、 通信协议规范

系统对通信信道进行了精准拆分，以兼顾流式体验和服务器连接成本。

### 1. 标准业务操作 (RESTful API)
- **适用场景**：所有轻量级、确定性的业务增删改查及状态扭转。
- **典型链路**：
  - 创建阅读/计划项目 (`POST /api/v1/projects`)
  - 触发项目归档与生成经验复盘 (`PUT /api/v1/projects/{id}/archive`)
  - 手动触发闲时图谱同步 (`POST /api/v1/knowledge/graph-sync`)
  - 提交沙箱技能审批入库 (`POST /api/v1/skills/{id}/approve`)

### 2. 伴读 Agent 对话流 (Server-Sent Events - SSE)
- **适用场景**：AI 导师启发的流式返回、打字机效果、后台深度编译 (Trace-to-Skill) 的进度推送。
- **选择缘由**：相较于双向 WebSocket，SSE 为单向流，极其适合大模型的逐字返回（Stream Token），且其底层复用 HTTP/1.1 或 HTTP/2 连接，更易穿透防火墙与负载均衡，降低了后端的长期连接维护成本。
- **典型链路**：
  - **对话请求**：前端通过 REST API (`POST /api/v1/agent/discuss`) 提交提问，后端返回 `202 Accepted` 或直接返回 Stream Response。
  - **流式接收**：前端利用 `EventSource` 或 Fetch ReadableStream 接收后端推送的数据块，在 UI 上实时渲染 Markdown 并自动滚动。

## 三、 核心业务流程的发起与协同契约

以下为系统中几种关键的协同流程及其控制权归属：

### 1. 划词写笔记与一键转存
- **发起方**：前端绝对主导。
- **协同契约**：前端拦截原生 `mouseup` 与划线事件，生成富文本临时卡片并获取物理页码/锚点（`source_anchor`）。前端防抖 500ms 后调用后端 `POST /api/v1/notes` 接口静默保存。后端必须无条件接收前端上报的物理锚点。

### 2. Trace-to-Skill 提炼漏斗
- **发起方**：前端触发，后端主导。
- **协同契约**：前端点击“一键提取本章精华”，向后端发送当前章节 ID。后端负责从数据库/缓存中反查该章节的所有原文切片与过往笔记，打包提交给大模型，生成 `SKILL.md` 后落盘至沙箱，最后通过长连接或短轮询通知前端编译完成。

### 3. 半自动重调度计算
- **发起方**：前端发起意图，前端/后端协同计算。
- **协同契约**：当任务逾期被前端拦截后，前端在日历/看板上发起拖拽。前端通过拓扑关联实时绘制“橡皮筋”联动预演效果；确认修改后，向后端发送批量更新请求。后端进行持久化并下发成功确认。

### 4. 数据供给关系与懒加载
- **图谱漫游渲染**：由于全局知识图谱可能包含上万节点，禁止前端一次性拉取。前端必须根据当前视窗的缩放级别与中心坐标，向后端发起分页/范围查询请求 (`GET /api/v1/graph/nodes?bbox=...`)。后端返回截断数量的核心节点，前端进行分批懒加载渲染。
- **融合阅读笔记供给**：打开阅读项目时，后端负责将“用户划词笔记”与“Agent 伴读聊天记录”在逻辑层整合并组装为统一实体格式 (Unified Note Card Model) 投递给前端，前端直接按时间线渲染为瀑布流，屏蔽底层的异构差异。
