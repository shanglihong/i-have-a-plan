## 二、 核心关键点共识

开发团队与 Lead 就系统竞品分析后的高阶架构、商业机会与 8 大模块技术/交互规格达成以下核心共识，作为后续系统设计的底层契约：

- **1. 最大核心差异化特色 (知行统一)**
  > 本系统打破了以 NotebookLM / Heptabase 为代表的“纯阅读输入端”与以 Taskade / Motion 为代表的“纯任务执行端”之间的物理断层，锁定“输入阅读提炼方法论 -> 编译为 YAML 步骤技能 -> 注入计划项目自动拆解 -> 执行任务指导行动”的全链路闭环作为核心破局杀手锏。
- **2. 知识输入侧 (模块 1-4) 的技术与交互规格共识**
  > * **项目初始化**：借鉴 Taskade 模板及 Agent 绑定模式，结合 Motion 截止时间等硬性限制输入，设计双轨项目初始化双栏引导向导。
  > * **文档解析**：吸收 Heptabase 的 PDF 物理大纲映射与 Readwise Reader 的 2px 蓝绿进度条上传逻辑，实现多格式解析、切片绑定与折叠大纲树渲染。
  > * **渐进式伴读**：参考 NotebookLM 的 65%/35% 左右分栏与 Monica 的划词浮动气泡，并在章节末尾 5% 范围引入弱挂起气泡提示。
  > * **混合知识库**：吸收 NotebookLM 的点击数字引用平滑滚动及原文脉冲闪烁动效，并借鉴 Heptabase 设计带文字描述的贝塞尔弹性物理双向连线。
- **3. 任务执行侧 (模块 5-8) 的技术与交互规格共识**
  > * **技能编译**：借鉴 Taskade 将自然语言指令编译为结构化 AI Agent 的思路，规范 Trace-to-Skill 编译 `SKILL.md` 任务流。
  > * **技能沙箱**：吸收 Heptabase 的 Section 物理隔离盒子与 Taskade 的步骤实时调试，设计独立沙箱卡片流编辑器，提供拖拽排序与依赖连线编辑。
  > * **计划注入**：借鉴 Taskade 计划应用 Agent 时的光流特效，配合大纲骨架屏渐进式分层渲染出结构化任务树。
  > * **任务链挂起**：借鉴 Motion 逾期冲突的挂起重调度，以及 Taskade 强勾选锁定任务时的卡片左右抖动与红光闪烁交互，作为本系统的防御底线。

## 三、 核心分歧与风险

在对标竞品底层逻辑并进行防御性博弈后，本阶段识别出系统在体验、计算成本和数据流稳定性方面的以下核心风险与架构分歧：

- **PA-01（体验与打扰冲突）**：借鉴 Monica 的浮窗交互，如果伴读 Agent 采用频繁弹窗会破坏用户沉浸式电子书阅读，导致严重打扰；而像 Heptabase 完全依赖人类手动高亮，又面临缺乏 AI 引导而流于随性的风险。
- **PA-02（运行成本分歧）**：Heptabase 因成本和复杂度不支持图谱自动构建，而普通的 Graph RAG 在每次划线时实时调用 LLM 构建实体关系会产生海量 Token 调用，面临极高昂的运行成本。
- **PA-03（大模型编译幻觉）**：Taskade 在自动生成工作流时完全依赖 LLM，若模型产生幻觉生成了步骤循环依赖，因缺乏前置阻断拦截校验，会导致整个任务模板彻底失效死锁。
- **PA-04（计划执行超时挂起）**：Motion 在计划冲突时拉起重调度，但长周期下若用户长时间离开，LLM 会话长连接不设超时会导致会话资源死锁；直接清除又易导致任务上下文丢失。

## 五、 决策建议

针对第三章中列出的核心风险与分歧，对比各竞品路线进行技术和业务折中，最终裁决如下：

### 1. P1 级裁决 (Lead 自决项)

#### 待审批项 PA-01：AI 伴读过程中的强打扰与用户阅读流畅度冲突
- **决策方案**：**方案 A（推荐，借鉴 NotebookLM/Monica 弱打扰气泡）**：左右分栏，选中文字浮现菜单一键 Discuss，仅当滚动至章节最后 5% 时在右侧渐显推荐气泡。对比**方案 B（替代，借鉴 Taskade 侧边栏强制对话并锁屏）**。
- **折中与依据**：放弃强制阻断答题，将滚动和翻页自主权完全还给用户（方案 A）。仅将输入思考作为“知识内化”的加分项，在最大限度保障极致沉浸式阅读的同时，实现伴读问答的收回，体验和自由度显著优于方案 B。
- **当前状态**：**已裁决并锁定**。

#### 待审批项 PA-02：混合知识库 (Graph RAG) 构建的运行 Token 成本
- **决策方案**：**方案 A（推荐，借鉴 NotebookLM 密集 RAG 缓存 + 闲时后台异步增量建图）**：平时伴读仅执行轻量级密集向量检索并写入临时缓存，归档时或手动触发同步图谱。对比**方案 B（替代，借鉴 Heptabase 纯人类手动建立卡片双链连线，或每次划线实时建图）**。
- **折中与依据**：放弃每次划线均实时触发 LLM 图谱分析的昂贵做法（方案 B）。将图谱生成挪到低频的后台闲时异步批处理（方案 A）。虽然这牺牲了关系图谱的即时更新性，但换取了运行 Token 消耗降低 80% 的高性价比优势。
- **当前状态**：**已裁决并锁定**。

#### 待审批项 PA-04：长周期计划执行中的状态丢失与 LLM 资源死锁
- **决策方案**：**方案 A（推荐，借鉴 Motion 逾期重调度 + 状态 Redis 持久化挂起）**：用户离开超过 24 小时，自动释放会话，状态写入持久缓存。登录时顶部毛玻璃提示，一键唤醒并以水波纹扫过刷新重载。对比**方案 B（替代，借鉴 Taskade 列表弱依赖且允许越权，或保持长连接无限挂起）**。
- **折中与依据**：放弃让服务器承担无休止长连接的方案（方案 B），采用超时优雅休眠并配合水波纹扫过的动效重新唤醒（方案 A）。接受了微小的主动重载延迟，换取服务器会话资源的高效释放，极大地防范了长周期下的死锁。
- **当前状态**：**已裁决并锁定**。

### 2. P2 级裁决 (人机共决项)

#### 待审批项 PA-03：大模型技能编译的幻觉、死循环依赖与执行死锁
- **决策方案**：**方案 A（推荐，借鉴 Taskade 实时调试 + Heptabase 的 Section 盒子物理拦截）**：沙箱临时隔离、拓扑排序校验，检测到环路依赖时前端卡片红色发光，连线变红抖动，锁定并变灰“批准入库”按钮。对比**方案 B（替代，传统纯文本直接保存后在运行时报错拦截）**。
- **折中与依据**：新编译技能处于 `skills/sandbox/` 物理隔离区，载入沙箱前执行拓扑排序。若检测到环路依赖，前端受影响的步骤卡片红色发光，连线变红抖动，锁定并变灰“批准入库”按钮。该决策已升级并由人类授权：必须由人类在沙箱中拖拽断开环路，纠正幻觉后方可入库。
- **当前状态**：**已通过升级机制由人类用户在交互中授权落定**。
